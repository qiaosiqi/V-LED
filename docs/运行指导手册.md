# V-LED 运行指导手册（教师汇报版）

> 适用项目：实验四——Linux 驱动程序
>
> 目标环境：Ubuntu 24.04.4、Linux 6.17.0-35-generic、x86_64、GCC 13.3；
> Windows 端使用 Python 3.12 和 Tkinter。
>
> 本手册用于向老师完整汇报 V-LED 的业务逻辑、功能项、运行方法和可观察结果。
> 所有关键命令都附有中文注释，可按第 6 节的顺序现场演示。

## 1. 项目一句话说明

V-LED 把 Linux 字符设备驱动、用户态命令工具、事件驱动 UDP bridge 和 Windows
图形模拟器连接成一条完整链路：用户通过 `/dev/vled` 写入控制命令，驱动维护虚拟
LED 状态并返回 JSON；`vled_bridge` 使用 `poll()` 等待状态变化，再把完整状态发送
到 Windows；Tkinter 模拟器校验报文后在主线程渲染文字、颜色、亮度和显示模式。

```text
vled_cli ──文本命令──> /dev/vled ──状态 JSON──> vled_bridge
                           │                         │
                           │ PAGE_SIZE / mutex       │ UDP 9000
                           │ per-open / poll          ▼
                           └──────────────────> Windows VLED 模拟器
```

内核模块不直接联网。网络发送放在 Linux 用户态，图形渲染放在 Windows 用户态，
三部分职责清晰，便于分别测试和定位问题。

|---|---|---|---|

| Linux 用户态 | `tools/vled_poll_probe` | 验证 poll、阻塞/非阻塞 read 和信号中断 | 证明事件驱动不是忙轮询 |
| Linux 用户态 | `tools/vled_verify.sh` | 构建、业务、边界、并发、poll、生命周期全回归 | 一条命令完成 P1–P5 验收 |

## 3. 核心业务逻辑

### 3.1 模块与设备生命周期

1. `vled_init()` 自动申请 major/minor，注册 `cdev`，创建 class 和 `/dev/vled`。
2. 模块加载完成时，驱动已经生成默认 version 0 状态 JSON，不会让第一次读取为空。
3. 每次 `open()` 分配一个独立 `vled_file_context`。
4. `release()` 只释放当前打开实例的快照和缓冲区，不影响其他进程。
5. `vled_exit()` 删除设备节点、class、cdev 和设备号，保证注册与清理对称。

默认状态为：

```json
{"type":"state","width":32,"height":16,"text":"","color":[255,255,255],"brightness":100,"mode":"static","version":0}
```

### 3.2 PAGE_SIZE、per-open 与稳定快照

- 驱动拥有一个 `PAGE_SIZE` 共享状态页，始终保存最新且完整的 JSON。
- 每个 `open()` 另有自己的 PAGE_SIZE 写入页、`write_offset`、`read_offset` 和读取快照。
- 一个 FD 分段读取期间，其他 FD 即使修改状态，旧 FD 仍读完同一旧版本 JSON，
  不会拼接新旧字段。
- 两个独立 FD 的读写偏移互不影响；`dup()` 出来的描述符按 POSIX 语义共享同一上下文。
- 设备不可 seek，`lseek()` 返回 `ESPIPE`。

### 3.3 写入、候选状态与原子回滚

一次 `write()` 代表一条完整命令。驱动先把本次数据复制到当前 FD 的写入页，再基于
旧状态构造候选状态和候选 JSON；只有解析、范围检查和 JSON 构建全部成功，才在
mutex 临界区一次性发布。任何失败都会清除本次片段，并保持共享状态、version、
读快照和写偏移不变。

主要边界：

| 场景 | 行为 |
|---|---|
| 零长度 read/write | 返回 0，状态和偏移不变 |
| 单次写入 `count >= PAGE_SIZE` | 返回 `EMSGSIZE` |
| 当前 FD 累计空间不足 | 返回 `ENOSPC` |
| 语法、参数、类型或范围错误 | 返回 `EINVAL` |
| `PIXEL` | 返回 `EOPNOTSUPP`，明确表示已识别但未实现 |
| 无新版本的非阻塞 read | 返回 `EAGAIN` |
| 阻塞 read 被信号打断 | 返回标准中断错误 |

### 3.4 version、poll 与 wait queue

- `version` 只在可观察字段实际改变时加 1。
- 重复设置相同值、`STATUS`、失败命令和空状态下再次 `CLEAR` 都不增加 version。
- 新 FD 可以立即读取当前状态。
- 一个长期打开的 FD 消费当前版本后，阻塞 read 会在 wait queue 等待下一版本；
  非阻塞 read 返回 `EAGAIN`。
- `.poll` 只在当前 FD 有未消费版本时返回 `POLLIN | POLLRDNORM`。
- 只有实际状态变化才调用 `wake_up_interruptible()`；无意义写入不会产生事件。

### 3.5 bridge 与 Windows 模拟器

`vled_bridge` 只打开一次设备，用 `poll()` 等待事件。读到状态后，它先验证 JSON 的
固定字段、类型和值域，再通过 IPv4 UDP 发送；超时只作为健康等待，不重复发送旧状态。
SIGINT/SIGTERM 会结束 poll、关闭设备 FD 和 socket。

Windows 后台线程只负责 UDP 接收、UTF-8/JSON/协议校验和投递不可变事件；Tk 主线程
每 20 ms 排空队列并整体替换状态。事件队列最多 1024 项，日志最多 200 项，避免高频
运行导致内存无界增长。非法报文只记录错误，不会部分更新界面。

手动颜色/亮度覆盖只影响显示值，后台仍保存最新 UDP 状态；点击恢复后立即采用最新值。
“本地预览清屏”只隐藏 Windows 预览文字，不会伪装成已经向 Linux 驱动写入 `CLEAR`。

## 4. 六条业务命令

| 命令 | 合法格式 | 业务效果 | version 规则 |
|---|---|---|---|
| `TEXT` | `TEXT <任意 UTF-8 文本>` | 设置显示文字；允许空文本、中文、空格和需 JSON 转义的字符 | 文字实际变化时 +1 |
| `COLOR` | `COLOR <r> <g> <b>` | 设置 RGB，三个通道范围均为 0–255 | 任一通道实际变化时 +1 |
| `BRIGHTNESS` | `BRIGHTNESS <0-100>` | 设置亮度百分比 | 数值实际变化时 +1 |
| `MODE` | `MODE static` 或 `MODE scroll` | 设置静态或滚动显示 | 模式实际变化时 +1 |
| `CLEAR` | `CLEAR` | 只清空 text，保留颜色、亮度和模式 | text 非空时 +1 |
| `STATUS` | `STATUS` | 不修改业务字段，用于刷新当前 FD 的读取快照 | 永不增加 |

输出 JSON 固定包含：`type`、`width`、`height`、`text`、`color`、`brightness`、
`mode` 和 `version`。Windows 端还会检查尺寸为正、单边不超过 128、总 LED 数不超过
4096、文字不超过 1023 UTF-8 字节、颜色/亮度/mode/version 都在合法范围。

## 5. 运行前准备

### 5.1 Windows 条件

- Python 3.12，且可导入 Tkinter。
- Windows VMnet8 地址在已验证环境中为 `192.168.57.1`。
- 防火墙允许 Python 接收 UDP 9000。

### 5.2 Linux 条件

- Ubuntu 24.04.4，运行内核 `6.17.0-35-generic`。
- 已安装与 `uname -r` 完全匹配的 headers、GCC、make 和 kmod。
- 当前用户可以通过 sudo 加载/卸载模块。
- 构建和普通工具使用普通用户运行；仅 `insmod`、`rmmod`、设备权限和受限内核日志需要 sudo。

## 6. 教师现场汇报流程

建议按“架构说明→Windows 启动→Linux 构建加载→六条命令→跨机演示→高级机制→
清理”的顺序汇报，全程约 8–12 分钟。

### 6.1 Windows：确认解释器并启动模拟器

在仓库根目录打开 PowerShell：

```powershell
# 查看 Python 版本，确认使用 Python 3.12。
python --version

# 验证当前解释器包含 Tkinter，并打印 Tk 版本。
python -c "import tkinter; print('Tkinter 可用，TkVersion=', tkinter.TkVersion)"

# 启动 Windows VLED 图形模拟器；窗口保持运行，等待 Linux UDP 状态。
python simulator/vled_sim.py
```

预期：窗口启动成功，日志显示监听 `0.0.0.0:9000`。此时可先做本机烟雾测试：

```powershell
# 从 Windows 本机向 UDP 9000 发送一条经过协议校验的状态报文。
python simulator/test_udp.py --text "Windows 本机测试" --color 0 128 255 --brightness 70 --mode static --version 1
```

预期：模拟器显示“Windows 本机测试”、蓝色系、70% 亮度和 static 模式。

### 6.2 Linux：确认仓库和目标环境

在 Linux 仓库根目录执行：

```bash
# 进入目标 Linux 上的 V-LED 仓库。
cd "$HOME/siqi_ws/V-LED"

# 查看当前分支和工作树，避免在错误分支演示。
git status --short --branch

# 打印当前提交，便于老师把运行结果与源码版本对应。
git rev-parse HEAD

# 查看运行内核；它必须与 headers 和模块 vermagic 对应。
uname -r

# 查看编译器版本，说明目标环境使用 GCC 13.3。
gcc --version | head -n 1

# 确认当前内核的构建目录存在。
ls -ld "/lib/modules/$(uname -r)/build"
```

### 6.3 Linux：从干净构建开始

```bash
# 如果旧 vled 模块仍在内核中，先卸载；未加载时忽略该情况。
sudo rmmod vled 2>/dev/null || true

# 清理旧驱动构建产物，保证本次从源码重新构建。
make -C driver clean

# 使用目标内核匹配的 GCC 13 构建驱动，并显示完整编译命令。
make -C driver V=1 CC=x86_64-linux-gnu-gcc-13

# 清理旧用户态可执行文件。
make -C tools clean

# 以警告视为错误的方式构建 CLI、bridge 和两个探针。
make -C tools CFLAGS='-Wall -Wextra -Werror -O2'
```

预期：生成 `driver/vled.ko`、`tools/vled_cli`、`tools/vled_bridge`、
`tools/vled_fd_probe` 和 `tools/vled_poll_probe`，构建过程无 warning/error。

### 6.4 Linux：加载模块并检查设备

```bash
# 把刚构建的内核模块动态加载到当前内核，无需重启系统。
sudo insmod driver/vled.ko

# 在隔离的课程实验机上临时允许普通用户读写设备。
sudo chmod 666 /dev/vled

# 查看自动创建的字符设备节点、权限和 major/minor。
ls -l /dev/vled

# 确认 vled 模块已经出现在内核模块列表中。
lsmod | grep '^vled '

# 查看模块作者、许可证、vermagic 等元信息。
modinfo driver/vled.ko
```

预期：`/dev/vled` 为字符设备，模块名为 `vled`，vermagic 与当前内核一致。

### 6.5 Linux：读取默认状态

```bash
# 新加载模块后直接读取默认 version 0 JSON。
./tools/vled_cli read /dev/vled
```

预期：得到完整且可解析的默认 JSON，text 为空、白色、亮度 100、static、version 0。

### 6.6 Linux：逐条演示六项业务命令

#### TEXT：文字

```bash
# 写入英文文字，实际改变 text，因此 version 增加 1。
./tools/vled_cli write "TEXT Hello VLED" /dev/vled

# 读取状态，确认 text 和 version 已更新。
./tools/vled_cli read /dev/vled

# 写入中文，证明 UTF-8 文本可以通过驱动、JSON、UDP 和 GUI。
./tools/vled_cli write "TEXT 中文演示" /dev/vled

# 再次读取，确认中文没有乱码。
./tools/vled_cli read /dev/vled
```

#### COLOR：颜色

```bash
# 把颜色设置为红色；RGB 三通道合法范围是 0–255。
./tools/vled_cli write "COLOR 255 0 0" /dev/vled

# 读取状态，确认 color 变为 [255,0,0]。
./tools/vled_cli read /dev/vled
```

#### BRIGHTNESS：亮度

```bash
# 把亮度设置为 80%；合法范围是 0–100。
./tools/vled_cli write "BRIGHTNESS 80" /dev/vled

# 读取状态，确认 brightness 为 80。
./tools/vled_cli read /dev/vled
```

#### MODE：静态/滚动模式

```bash
# 切换为滚动显示模式；合法值只有 static 和 scroll。
./tools/vled_cli write "MODE scroll" /dev/vled

# 读取状态，确认 mode 为 scroll。
./tools/vled_cli read /dev/vled

# 切回静态显示模式。
./tools/vled_cli write "MODE static" /dev/vled
```

#### STATUS：读取刷新但不修改 version

```bash
# 执行 STATUS；命令成功，但不改变业务字段和 version。
./tools/vled_cli write "STATUS" /dev/vled

# 读取当前完整状态，说明 STATUS 不会制造虚假版本。
./tools/vled_cli read /dev/vled
```

#### CLEAR：只清文字

```bash
# 清空文字；颜色、亮度和模式保持不变。
./tools/vled_cli write "CLEAR" /dev/vled

# 读取状态，确认 text 为空，其他字段仍保留。
./tools/vled_cli read /dev/vled
```

### 6.7 Linux→Windows 一键业务演示

确认 Windows 模拟器仍在运行，然后在 Linux 执行：

```bash
# 启动事件驱动 bridge，并依次演示文字、颜色、亮度、模式、中文和清屏。
./tools/vled_demo.sh 192.168.57.1 /dev/vled
```

脚本内部会依次执行：英文文字、红色、80% 亮度、scroll、中文、青色、static、CLEAR。
每一步都会打印命令和当前 JSON；任一步失败，脚本会立即停止。正常结束时出现：

```text
Demo complete; stopping bridge cleanly.
```

Windows GUI 应同步显示上述变化，并在日志中看到来自 Linux 的 UDP 状态。

### 6.8 可选：单独展示事件驱动 bridge

如果老师希望观察 bridge 的长期运行行为，可在 Linux 单独启动：

```bash
# 长期打开 /dev/vled，用 poll 等待变化，并把状态发送到 Windows UDP 9000。
./tools/vled_bridge 192.168.57.1 9000 /dev/vled 200
```

另开 Linux 终端执行任意有效写入，bridge 只在新版本出现时打印 `sent ...`；没有变化
时不会重复发送旧状态。演示结束按 `Ctrl+C`，预期打印 `UDP bridge stopped`。

### 6.9 可选：展示高级机制和自动验收

`vled_fd_probe` 会检查默认 version 0，因此先重新加载一个全新模块实例；为使两次
演示互不影响，poll 探针前也再次重新加载：

```bash
# 卸载业务演示使用的模块，准备 version 0 的全新实例。
sudo rmmod vled

# 重新加载刚构建的驱动，初始化默认状态。
sudo insmod driver/vled.ko

# 恢复课程实验机上的普通用户访问权限。
sudo chmod 666 /dev/vled

# 验证 PAGE_SIZE、per-open 读写偏移、稳定快照、JSON 转义和失败回滚。
./tools/vled_fd_probe /dev/vled

# 卸载被 P1 探针修改过状态的模块。
sudo rmmod vled

# 再加载一个全新模块实例，隔离 poll 探针的输入状态。
sudo insmod driver/vled.ko

# 恢复课程实验机上的普通用户访问权限。
sudo chmod 666 /dev/vled

# 验证 poll、wait queue、阻塞/非阻塞 read、信号中断和高频最终状态。
./tools/vled_poll_probe /dev/vled
```

预期：分别出现 `VLED P1 probe: all checks passed` 和
`VLED P5 probe: T-POLL-01..07 passed`。

完整自动验收会自行构建、加载、测试并卸载，因此运行前先卸载当前模块：

```bash
# 卸载当前模块，避免自动验收拒绝替换已加载实例。
sudo rmmod vled

# 运行 P1–P5 完整验收；脚本内部按需调用 sudo，不要给整个脚本加 sudo。
./tools/vled_verify.sh
```

预期末行：

```text
VLED P1-P5 verify: PASS (module will be unloaded by cleanup trap)
```

### 6.10 演示结束后清理

如果前一步没有运行完整验收，而模块仍加载，则执行：

```bash
# 卸载 vled 模块，触发驱动清理路径。
sudo rmmod vled

# 确认模块已经从内核模块列表中消失。
lsmod | grep '^vled' || echo 'vled 模块已卸载'

# 确认自动创建设备节点已经删除。
test ! -e /dev/vled && echo '/dev/vled 已删除'

# 查看最近内核日志，确认只有正常注册/注销消息。
sudo journalctl -k -n 50 --no-pager | grep -i vled
```

## 7. 错误路径演示

这些命令用于说明驱动会拒绝错误输入，而且失败不会改变状态或 version。
建议安排在第 6.6 节业务命令之后、第 6.10 节卸载清理之前，确保 `/dev/vled` 已加载
且普通用户可写。

```bash
# 演示颜色越界；256 超出 RGB 上限，应返回 Invalid argument。
./tools/vled_cli write "COLOR 256 0 0" /dev/vled

# 演示亮度越界；101 超出上限，应返回 Invalid argument。
./tools/vled_cli write "BRIGHTNESS 101" /dev/vled

# 演示非法模式；blink 不属于 static/scroll，应返回 Invalid argument。
./tools/vled_cli write "MODE blink" /dev/vled

# 演示已识别但未实现的 PIXEL，应返回 Operation not supported。
./tools/vled_cli write "PIXEL 0 0 255 255 255" /dev/vled

# 演示未知命令，应返回 Invalid argument。
./tools/vled_cli write "UNKNOWN" /dev/vled

# 错误写入后重新读取，确认状态和 version 没有被部分修改。
./tools/vled_cli read /dev/vled
```

注意：这些是预期失败命令，CLI 返回非零退出码是正确现象。

## 8. Windows 界面功能说明

| 界面功能 | 实际语义 |
|---|---|
| UDP 状态显示 | 显示最近一次通过完整协议校验的 Linux 状态 |
| 手动颜色 | 只覆盖当前显示颜色，不覆盖后台最新 UDP color |
| 恢复 UDP 颜色 | 立即采用覆盖期间收到的最新 UDP color |
| 手动亮度 | 只覆盖当前显示亮度，不覆盖后台最新 UDP brightness |
| 恢复 UDP 亮度 | 立即采用覆盖期间收到的最新 UDP brightness |
| 本地预览清屏 | 只隐藏 Windows 文字；不是 Linux 驱动 CLEAR |
| static | 文字居中静态显示 |
| scroll | 文字按渲染循环滚动显示 |
| 日志清空 | 只清 GUI 日志，不影响设备状态 |
| 关闭窗口 | 设置停止事件、关闭 socket、等待接收线程退出 |

## 9. 老师常问问题与回答要点

### 为什么用字符设备？

实验要求实现 open/read/write/close、设备号、cdev 和设备节点。字符设备能让用户态
通过标准文件接口控制虚拟 LED，同时把内核机制和业务协议清晰结合。

### 为什么内核不直接发送 UDP？

内核只负责设备抽象和状态一致性，网络转发交给用户态更安全、易调试，也避免在内核
中引入 socket、重传和网络配置复杂度。

### 怎样证明多进程偏移独立？

每次 `open()` 都分配独立上下文；`vled_fd_probe` 同时打开多个 FD，验证各自从偏移
0 读取、各自拥有写容量，并证明一个 FD 的分段快照不会被另一个 FD 的更新拼接。

### 怎样防止读到半个状态？

写入先生成候选状态和完整候选 JSON，成功后才在 mutex 内整体替换共享状态页；读取
则先捕获稳定快照，再按当前 FD 的偏移分段返回。

### 为什么重复命令不唤醒 poll？

事件表示“可观察状态发生变化”。重复值、STATUS 和失败命令都没有新版本，若仍唤醒
会造成虚假事件和重复 UDP 发包，所以只有 `changed=true` 才递增 version 并唤醒。

### Tkinter 为什么不会跨线程崩溃？

UDP 后台线程完全不调用 Tk 控件，只把不可变事件放入线程安全队列；所有界面修改都由
Tk 主线程通过 `root.after()` 完成。

### UDP 不保证送达，项目怎样说明这个边界？

bridge 能确认本地 `sendto()` 是否成功，但 UDP 没有应用层确认，不能仅凭发送日志证明
Windows 已显示。现场以 Windows GUI 状态和接收日志作为端到端结果。
